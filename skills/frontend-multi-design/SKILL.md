# Frontend Multi Design

## 核心规则

编写全面的实施计划，假设工程师是一位熟练的开发者，但对我们的工具集或问题领域几乎一无所知。记录他们需要了解的所有信息：每个任务需要修改哪些文件，代码，测试，他们可能需要检查的文档，如何进行测试。将整个计划拆分为小块任务，便于执行。循“开闭原则”（DRY）、“尽可能早地发现问题”（YAGNI）和“测试驱动开发”（TDD）。

## 工作流程

1. 在开始修改代码前，请先阅读项目的基础工程和基础模块的相关文件。
   - 阅读了解路由、存储、API模块、公共工具、可能会使用的公共组件。
   - 识别本地文件中的框架、包管理器、UI 库、代码检查规则、路径别名以及现有的代码风格。

2. 确定制定计划的子智能体个数、责任
   - 根据任务选择需要开启的子智能体个数，如无必要不要开启多余的子智能体。
   - 子智能体来制定修改计划，主智能体负责审核计划。如果计划审核未通过，子智能体继续修改计划直至审核通过。

3. 子智能体制定计划原则
   - 请尽可能的使用公共模块。并且再计划中详细标注，使用完整文件路径。
   - 设计具有清晰边界和明确的单元。每个文件应承担一个明确的责任。
   - 你最好在能理解其上下文的代码的时候进行推理。专注当前修改的文件，让修改更可靠。
   - 比起功能过多的大文件，更倾向于使用小而专注的文件。
   - 需要一起变更的文件应放在一起。按责任、模块划分，而不是按技术层级划分。
   - 在现有代码库中，遵循已有的模式。如果代码库使用了大文件，不要单方面重新结构化；但如果某个你正在修改的文件变得难以管理，那么在计划中加入拆分是合理的。

4. 将验证后的计划写入 docs/design/YYYY-MM-DD-<topic>-design目录下
   - 主文件名称是design-plan.md，主文件中对各个模块、任务的具体实现放在各个子模块文件中，主文件主要是对任务、模块的描述。
   - 各个模块的具体描述、实现过程、完整代码修改请放在各个子模块的文件中。
   - 各个模块写入docs/design/YYYY-MM-DD-<topic>-design/module/目录下，文件名称是<module>-design.md

## 具体代码实现

当计划开始需要编写具体代码的实现时，请使用references/code-creator.md文件

## 注意

- 始终使用完整的文件路径。
- 每一步都提供完整代码——如果某一步更改了代码，请展示该代码
- 精确命令及预期输出。
- DRY、YAGNI、TDD。
- 在能够完成任务的前提下，保证计划是最小程度的修改。
- 不要修改公共模块，如果功能需要对公共模块修改，请复制到使用模块的内部后再修改。
