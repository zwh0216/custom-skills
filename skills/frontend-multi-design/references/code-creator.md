# Code Creator

当涉及实现具体的前端代码，使用此文件。

## 核心规则

- 让状态保持可预测性：只使用一个来源的数据，其他所有内容都以此为基础进行推导。
- 明确数据流的流向：保持数据流向自上而下
- 使用小型、专注的组件：这样更容易进行测试、复用和维护。
- 避免不必要的重新渲染：合理使用计算属性和观察器。
- 可读性很重要：编写清晰、易于理解的代码。请尽可能的添加中文注释。

## 1)确认组件的架构设计

- 项目如果使用了typescript，请使用`<script setup lang="ts">`。
- 项目如果是vue3，请使用Vue 3的组合式API。

## 2)使用功能文件夹模式进行模块布局

- 页面入口组件作为index.vue放在模块根目录。
- 组件文件放在components/目录下，使用`<feature>.vue` 命名。
- 独属当前组件的特殊工具函数放在utils/目录下，使用`<feature>Utils.ts`命名。
- 独属当前组件的hooks放在hooks/目录下，使用`use<feature>.ts`命名。
- 如果当前架构使用了typescript，组件使用的ts类型（包括props，emits的类型）请放在types/目录下，使用`<feature>.type.ts`命名。

## 3)保持组件单一责任、专注

- 用一句话描述组件的单一职责。
- 实现组件前优先给组件定义相应的 props 和 emits。
- 默认情况下，将入口页、根页面以及路由级别视图组件都设置为组合式界面。
- 将功能界面和逻辑代码移出入口/根视图组件，除非该任务确实只是一个小型的单文件演示。

1. 当某个组件具有多个职责时，应该将其拆分开来处理这些职责
   - 更倾向于使用较小的组件和组合体，而不是一个“大型组件”。
   - 将状态管理/副作用处理功能整合到hooks中（ useXxx() ）。
   - 将用UI视图组件放入子组件中（将相关信息作为 prop 传递，事件则作为 event 传递）。
2. 当以下任何条件满足时，就拆分该组件。
   - 它拥有 3 个以上的独立用户界面区域（例如：表单、筛选器、列表、页脚/状态栏）。
   - 它为多个部分提供了完整的编排/状态以及丰富的呈现标记。
   - 组件可以重复使用，或可变为可复用的元素（如项目行、卡片、列表条目）。
3. Entry/root 视图和路由视图规则：
   - 保持入口/根部和路由视图组件的轻量化
   - 当某个功能包含独立部分时，不要将完整功能实现放在 entry/root/view 组件中。
   - 对于CRUD/列表功能（如待办事项、表格、目录、收件箱），至少应拆分为以下部分：
     - 功能容器组件
     - 输入/表单组件
     - 列表（和/或项目）组件
     - 页脚/操作或筛选/状态组件  
   - 仅在极小的临时演示中允许单文件实现；若选择此方式，需明确说明为何无需拆分。

## 4)组件的数据流

1. 保持数据流的单项传递，采用“向下使用 props，向上使用事件”作为主要模式。
2. 仅在真正双向的组件约定中使用 v-model。
3. 根据需要，通过 defineProps、defineEmits 和 InjectionKey 明确并类型化组件约定。类型请按照`2)使用功能文件夹模式进行模块布局`放置在指定目录文件中。
4. 使用 provide/inject 的相关组件需要特殊标注说明原因，并且provide只能提供readonly的只读数据。如果需要修改元数据，请通过provide提供action方法。

## 5)组合式api使用

1. 当逻辑被重复使用、具有状态或副作用较重时，应将其提取为可组合组件。
2. 保持可组合 API 小型化、类型化且可预测。
3. 将功能逻辑与呈现组件分离。

## 6)注释

尽可能的添加中文注释:

1. 函数、复杂组件、ts类型及其每个字段、定义的每个state需要添加中文注释，用于表达其作用。
2. 函数中的复杂步骤需要针对其步骤添加中文注释用于描述其步骤作用。

## 7)自我检查

1. 核心功能正常运行且符合需求。
2. 所有必读的参考文献均已阅读并应用。
3. 响应式模型简洁且可预测。  
4. 遵循了SFC结构和模板规则。  
5. 组件聚焦且结构良好，必要时进行拆分。  
6. 除非有明确的小型演示（small-demo）例外，否则 entry/root/route 组件仍保持为组合表面。  
7. 组件拆分决策明确且可辩驳（责任边界清晰）。  
8. 数据流契约明确且类型化。  
9. 在重用性或复杂性合理的情况下使用组合函数。  
10. 如果适用，state/side effects 被移至组合函数中。  
11. 可选功能仅在需求要求时才使用。  
12. 性能调整仅在功能完成之后实施。
